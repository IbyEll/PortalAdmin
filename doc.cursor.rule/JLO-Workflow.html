JLO-Workflow

.cursor/rules/JLO-Workflow.mdc · alwaysApply: true · Alias overlay JustLastOne

Regola alias della canonica ADMIN-Workflow per JustLastOne / JLO. Include step 3 veve su ogni procedi / gogo JLO-xxx.

Aggiornato 2026-06-22 — veve in pipeline workflow step 3.

Scopo

Applica lo stesso protocollo di ADMIN-Workflow quando l'istanza portal ha:

ParametroValore JLO
PRJ_NAMEJustLastOne
PRODUCT_REPO_PATHCheckout JustLastOne (non solo PortalAdmin host)
JIRA_KEY / PRJ_JIRA_PREFIXJLO
BranchSTORY---JLO-{num}-{slug} (idem BUG/TODO)

Trigger

Comando esempioNote
gogo JLO-507Stop se istanza non allineata a JustLastOne
procedi Story JLO-930Prefisso deve coincidere con PRJ_JIRA_PREFIX
chiudi fast Story JLO-930Chiusura senza gap test

Applicazione step-by-step

  1. Verifica istanza — PRJ_NAME=JustLastOne e PRODUCT_REPO_PATH punta al repo JustLastOne.
  2. Segui ADMIN-Workflow — step 1 branch, 2 pre-flight, 3 veve, 4 Step 0, 5 implementazione, 7 chiudi.
  3. Aree gap analysis — apps/, packages/, testScript/ nel product repo.
  4. Output END — link repo da PRJ_GITHUB_OWNER / PRJ_GITHUB_REPO overlay JustLastOne.

Ticket ADMIN nello stesso host

Per ticket ADMIN-xxx vale la stessa regola canonica con overlay AdminDashBoard e repo workflow PortalAdmin (product = stesso repo dell'host).

Vietato

  • gogo JLO-xxx con solo overlay AdminDashBoard e product non puntato a JustLastOne
  • Assumere che il repo workflow sia sempre PortalAdmin
In sintesi: JLO-Workflow non duplica la logica — è un puntatore contestuale a ADMIN-Workflow con parametri JustLastOne/JLO.
