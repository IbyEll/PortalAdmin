ADMIN-Comcom

.cursor/rules/ADMIN-Comcom.mdc · alwaysApply: true

Comando comcom nomefile — aggiunge commenti uniformi (testata + corpo) usando i template in .cursor/templates/comments/. Modifica solo commenti, mai la logica.

Scopo

Standardizzare la documentazione inline nel codice PortalAdmin: metadati, descrizione funzionale, generalizzazione, step numerati e JSDoc — con validazione obbligatoria prima della chiusura.

Non attiva workflow JLO/ADMIN (procedi, gogo, chiudi, veve).

Trigger

ComandoEffetto
comcom nomefileCommenta il file indicato (path relativo o assoluto)
comcom path/to/file.mjsIdem
Creazione entrypoint in scopeStesse regole anche senza comando esplicito

Scope automatico: entrypoint in runner/, scripts/, cruscotto.database/script_seed/, admin.portal.testscript/.

Sequenza agente (7 passi)

  1. Leggi file + consumer — file target e 1–2 import consumer (grep). Se esiste creato il: → conserva data e by (re-comcom).
  2. Scegli template — tabella tipi in .cursor/templates/comments/README.md.
  3. Leggi template — obbligatorio con Read prima di modificare il target. Se dubbio → template_default.mjs.
  4. Applica sul target — compila sezioni reali; mantieni ordine template; riga --- (120 trattini) prima di */; step // N. sul corpo.
  5. Valida — checklist V1–V14 (sotto); correggi finché tutti passano.
  6. Solo commenti — nessun refactor, rinomina o cambio logica; indentazione invariata.
  7. Output chat — una riga: template {nome} + {path} (no dump).

Validazione commento (V1–V14)

Controlli automatici

#Controllo
V1Zero placeholder: sostituire, compilare, esempio —, yyyy-mm-dd
V2Zero {…} in commenti testata/step
V3Zero glob * nel testo commento
V4Separatori --- = esattamente 120 trattini
V5Ultima riga testata = --- prima di */
V6Righe commento ≤ 120 caratteri
V7A capo solo su spazio (no parole spezzate)

Controlli strutturali

#Controllo
V8Metadati completi (tipo, creato, modificato)
V9Perché esiste / A cosa serve con bullet
V10Generalizzazione + Input
V11Re-comcom: creato invariato
V12Step corpo come da tipo template
V13Italiano (salvo termini tecnici)
V14Nessuna modifica logica eseguibile

Se un controllo fallisce: correggi → ripeti → solo allora output chat.

Vietato

  • Template non letto prima della modifica
  • Placeholder {…} nel file target
  • Modifica logica o frontmatter regola
  • Sovrascrivere creato il su re-comcom
  • Alterare righe *! (intoccabili)
  • Chiudere senza validazione V1–V14

Priorità

Su comcom e script in scope, questa regola prevale su «pochi commenti» (branch/exit, commenti per fase, ordine numerato inclusi).

In sintesi: comcom = leggi template → applica struttura → valida V1–V14 → una riga in chat. Esempi reali: cruscotto.database/cruscotto.db.config.mjs, cruscotto.database/script_seed/script_seed-lib.mjs.
