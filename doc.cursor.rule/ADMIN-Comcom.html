- Se la testata ha già creato il: → estrai e conserva data/ora e by (vedi § Re-comcom sotto).

  1. Scegli il template — tabella in .cursor/templates/comments/README.md.
  2. Leggi il template scelto con il tool Read — obbligatorio, prima di modificare il target:

Scope script eseguibili: oltre a comcom esplicito, quando crei o modifichi entrypoint in runner/, scripts/, cruscotto.database/script_seed/, admin.portal.testscript/ — stesse regole testata + corpo sotto (anche senza invocare comcom in chat, se l’utente chiede commenti su quello script).

  1. Applica sul file target la struttura del template letto:

- compila ogni sezione con contenuto reale (consumer, path, flag, export, Generalizzazione, Input) derivato da codice + grep; - mantieni sezioni, ordine e bullet del template (righe vuote, - sotto Perché/A cosa serve/Generalizzazione/Input/Consumatori/Export); - includi solo sezioni facoltative del template se pertinenti (Uso, Flag CLI, env, Route, Prerequisiti); - chiudi la testata con riga --- (120 trattini) prima di */; - aggiungi step // N. e JSDoc sul corpo come nel template (main, handler, export — non solo testata); - non inventare un formato alternativo.

  1. Valida commento — checklist § Validazione commento (obbligatoria; correggere prima di chiudere).
  2. Solo commenti — nessun refactor, rinomina o cambio logica; indentazione invariata.
  3. Output chat — una riga: template {nome} + {path} (no dump).

Validazione commento (passo 5 — obbligatorio)

Dopo aver applicato il template, rileggere la testata e i commenti aggiunti; non chiudere finché tutti i controlli passano.

Se un controllo fallisce

  3. Solo allora passo 7 (output chat).

Non segnalare comcom completato con testata incompleta o placeholder residui.

Formattazione (anti-righe-vuote)

  • Preferire patch mirate (StrReplace) sulla testata e sui blocchi senza commenti; riscrittura intera solo se necessario.
  • Su Write intero file: fine riga \n (LF) — non \r\n ripetuto riga per riga (su Windows può raddoppiare le righe in editor).

Cosa deriva dal template (non riscrivere in chat)

Tutto ciò che è nel file template letto al passo 3:

  • blocchi testata /* … / (titolo, descrizione funzionale a bullet, Generalizzazione + Input, uso/route, flag, env, consumatori, export);
  • formato step // N. {cosa} — {perché};
  • JSDoc su export e funzioni pubbliche non banali;
  • checklist implicita del template (es. help → exit 0 nello script-entrypoint).

La regola non duplica il template: se c'è conflitto, vince il template aggiornato in .cursor/templates/comments/.

Regole aggiuntive (oltre al template)

Descrizione funzionale (formato obbligatorio)

 * Descrizione funzionale:
 *
 *   Perché esiste:
 *   - …
 *
 *   A cosa serve:
 *   - …

Mai prosa inline su una riga (Perché esiste: testo lungo… senza bullet).

Testata metadati (righe ---)

Struttura obbligatoria subito dopo /** (JSDoc) o dopo la riga schema <!-- (HTML):

 * ---- 120 trattini ----
 * ** {TIPO} ** -- commentato il: …    
 * ---- 120 trattini ----
 * creato     il: …   by: …
 * modificato il: …   by: …
 * ticket refirement: …   (opzionale)
 * ---- 120 trattini ----
  • Separatori: esattamente 120 trattini dopo * (JSDoc) o 120 trattini a inizio riga (HTML).
  • Riga tipo: allineata a sinistra subito dopo * (JSDoc) o a inizio riga (HTML) — senza padding di centratura.
  • Vietato: creato/modificato senza riga --- sopra (sotto la riga tipo).

Esempio (lib):

 * ------------------------------------------------------------------------------------------------------------------------
 * ** LIBRARY MODULE ** -- commentato il: 2026-06-18 03:20
 * ------------------------------------------------------------------------------------------------------------------------
 * creato     il: 2026-06-17 00:00   by: IbyEll
 * modificato il: 2026-06-18 03:20   by: IbyEll
 * ------------------------------------------------------------------------------------------------------------------------

Dopo l’ultima sezione di testata (Prerequisiti, Variabili d’ambiente, Dipendenze, Export, …) — sempre una riga chiusura prima di */:

 * ------------------------------------------------------------------------------------------------------------------------
 */
  • Obbligatoria in ogni comcom (lib, script, app-module, page-html, page-styles, default).
  • Vietato chiudere la testata con */ subito dopo l’ultimo bullet senza riga --- finale.

Corpo codice — step // N. e JSDoc

Dopo la testata, commentare il flusso esecutivo (non import/export ovvi):

TipoObbligo

script-entrypointStep numerati // 1. … — … su main, validazione argv, fasi I/O, exit
app-module (server)Step su main, routing (handleRequest / handler API), shutdown; JSDoc su helper non banali

Formato step JS: // N. {cosa} — {perché} oppure // Step N — {cosa} — {perché} (italiano). Formato sezione CSS: / --- {nome} — {scopo} --- /. Regole .mdc: blocco meta <!-- … --> + sezioni ## (vedi template cursor-rule.mdc).

Non omettere step solo perché la testata è completa — comcom = testata + corpo commentato.

Titolo (blocco *****************************************************************)

  • Larghezza banda: 120 caratteri (come la riga di * separatore).
  • Max 120 caratteri per riga di titolo (solo testo titolo, senza prefisso né padding).
  • Se supera 120 caratteri → più righe nel blocco stellato; a capo solo su spazio — mai spezzare una parola.
  • Centratura: ogni riga titolo è centrata nella banda da 120 con spazi dopo il prefisso:

- JSDoc (/* /): + spazi + testo — spazi = floor((120 − lunghezza testo) / 2) - HTML (<!-- -->): + spazi + testo — spazi = floor((120 − 1 − lunghezza testo) / 2) (il * iniziale conta 1)

  • Ogni riga di un titolo multiriga è centrata in modo indipendente.

Esempio (lib/script):

 * ************************************************************************************************************************
 *              Ticket correlati backlog — parsing ADF description e issue link Jira.
 * ************************************************************************************************************************

Testo (tutte le righe commento, escluso titolo)

  • Max 120 caratteri per riga di testo (bullet, Generalizzazione, Input, Consumatori, Export, Uso, env, step // N., JSDoc in testata).
  • Escluso il blocco titolo stellato (max 120 caratteri/riga + centratura in banda 120).
  • Se supera 120 caratteri → a capo su spazio — mai spezzare una parola.
  • Bullet multiriga: riga successiva con stesso indent del testo bullet ( * in JSDoc, in HTML dopo -).

Esempio:

 *   - Il backlog arricchisce ogni issue con relatedKeys per tree e inspect; Jira espone
 *     description in ADF e issueLinks, non un campo flat.

Generalizzazione (obbligatoria in testata)

Dopo A cosa serve, sempre:

 * Generalizzazione:
 *   Si — …   oppure   No — …
 *
 * Input (obbligatorio se Si; se No: «Input: —»):
 *   - nomeIngresso — origine e ruolo (env, argv, overlay, bootstrap, API, …)
  • Si — modulo/script/pagina riusabile su più PROJECT_* / overlay, o comportamento guidato da parametri esterni (non hardcoded su un solo product name).
  • No — logica dedicata; nessun overlay o parametrizzazione rilevante.
  • Input — solo ingressi esterni al file (non variabili interne): PRJ_NAME, PRODUCT_REPO_PATH, JIRA_PROJECT_KEYS, flag CLI, window.CRUSCOTTO_PROJECT, query/hash, body HTTP, ecc.

Re-comcom — preserva creato il / by

Situazionecreato il / by (riga creato)commentato il / modificato il
Primo comcom (nessuna riga creato il:)Data/ora corrente + by: da git config user.name (se assente: by: —)Stessa sessione: uguali a creato
Re-comcom (riga creato il: già presente)Invariati — copia letterale dal file, mai riscrivereAggiorna a data/ora corrente; modificato il + by: (git user o —)
  • Aggiorna anche commentato il al passaggio corrente (re-comcom = refresh testata).
  • ticket refinement: — preserva se già presente; compila solo al primo comcom se noto dal contesto ticket.
  • Vietato su re-comcom: azzerare, spostare o sostituire creato il / by della riga creato.

Commenti *! (intoccabili)

Righe che iniziano con ! in / … / o //:

  • non modificare, rimuovere, compattare né tradurre;
  • comcom può aggiungere testata/step solo fuori da blocchi *! o sopra/sotto senza alterarli.

Vietato

  • ❌ Testata senza riga --- finale (120 trattini) prima di */
  • ❌ Corpo senza step // N. / // Step N — dove il template prevede flusso (script, server main, export lib, testscript)
  • ❌ Script entrypoint senza commento su process.exit / early return / flag CLI non banali
  • ❌ Sovrascrivere creato il / by (riga creato) su re-comcom
  • ❌ Applicare comcom senza aver letto un template (passo 3)
  • ❌ Formato libero o testata “a parole” senza struttura template
  • ❌ Lasciare {…} o sezioni schema nel file target
  • ❌ Modificare comportamento del file target (logica regola, frontmatter YAML description/globs salvo compilazione reale da contesto)
  • ❌ Inglese nei commenti (salvo termini tecnici: CLI, env, path)
  • ❌ Chiudere comcom senza passo 5 Validazione commento

Priorità

Su comcom e su script in scope (runner/, scripts/, cruscotto.database/script_seed/, admin.portal.testscript/), questa regola prevale su «pochi commenti» (branch/exit, commenti per fase, ordine numerato inclusi).
