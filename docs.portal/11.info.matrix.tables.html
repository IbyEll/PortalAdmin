Le matrici PortalAdmin (avanzamento / gap / audit e copertura test) persistono su SQLite cruscotto nelle tabelle matrix_*. Lo scan repo propone delta; il DB è la sorgente di verità delle righe; l’HTML è un artefatto di visualizzazione rigenerabile da query DB.

In una riga: ogni finding_id stabile (es. bug-tree-regenerate) è una riga in matrix_row per matrix_kind; ad ogni scan si crea un matrix_run, si upsertano le righe, si appendono eventi, si genera HTML da matrix.render.mjs.

1. Visione funzionale

Le matrici rispondono a domande diverse sul repo PortalAdmin:

DomandaMatriceEsempi di righe
Dove siamo con la migrazione? Cosa manca? C’è drift o ridondanza? portal_gap Architettura OK, gap PARKING live, audit R1–R7, bug import mancanti
Quali feature/API hanno test? Dove servono smoke o testscript? test_coverage Smoke CI coperto, portal-api parziale, gap su /issue.html

Per ogni riga l’operatore può:

  • consultare severità, stato repo, path coinvolti;
  • collegare un ticket Jira con il pulsante [Matrix GAP] o [Matrix TEST];
  • rigenerare la matrice dopo fix nel codice (scan aggiorna status / obsoleto).

matrix_run — snapshot scan

Ogni rigenerazione (CLI, API o toolbar) crea un run con:

  • matrix_kind — quale matrice;
  • source — es. matrix.portal.gap.analysis, api.matrix.regenerate;
  • metrics_json — sintesi numerica del run (arch %, gap, bug, …);
  • sync_run_id — opzionale, legame al sync Jira contemporaneo.

matrix_row — stato corrente finding

Chiave composta (finding_id, matrix_kind). Campi principali:

CampoSignificato
finding_idId stabile scan/catalogo (es. cov-gap-issue-html, audit-prio-r3)
section_idSezione UI (es. op-gap, audit-redundancy, orch)
statusgap · parziale · fatto · coperto · obsoleto · …
sevP0–P5 · info · warn
projectPortalAdmin o PortalAdmin.Cruscotto
voce / dettaglioTitolo riga e descrizione gap/stato repo
paths_jsonArray path file coinvolti
content_sigHash JSON status+sev+voce+dettaglio+paths — rileva delta scan
resolved_noteNota chiusura (es. ticket Jira collegato)

Regola: le righe non vengono mai DELETE; se assenti dallo scan → status = obsoleto.

matrix_row_event — audit trail append-only

Tipi evento (matrix.db.mjs):

  • scan_detected — prima occorrenza finding;
  • status_change — transizione stato (es. gap → fatto);
  • marked_obsolete — non più rilevato dallo scan;
  • issue_linked / issue_resolved — collegamento Jira.

matrix_finding_issue — link Crea → Jira

Sostituisce il legacy matrix.finding-issues.json. Una riga per (finding_id, matrix_kind) con jira_key univoco. FK logica verso jira_issue (stessa key nel cache backlog).

matrix_catalog — definizioni stabili

Seed da matrix.catalog.seed.mjs / catalogo audit (AUDIT_CATALOG). Definisce finding noti, sezione default e mapping scan id → audit id per la matrice gap.

Sequenza rigenerazione (portal_gap)

  1. analyzePortalAdvancement — grep repo, checklist architettura, finding catalogati;
  2. buildUnifiedMatrixSections — sezioni op-* (operativo) + audit-* (audit);
  3. persistUnifiedMatrixSections — run + upsert + obsoleto automatico;
  4. loadUnifiedMatrixSectionsFromDb — righe arricchite con link Jira;
  5. renderMatrixPage o merge incrementale (matrix.refresh.mjs) su HTML esistente.

7. Cosa viene visualizzato in HTML

Renderer: docs.portal.lib/matrix.render.mjs · stili: docs.portal/docs.style.css.

Struttura pagina

  • Header — titolo, meta (data run, link correlati), lead descrittivo;
  • Card metriche (data-adv-section="metrics") — conteggi gap, parziali, fatto, arch %;
  • Sezioni collapsabili (details.adv-card) — una per section_id;
  • Part heading — «Avanzamento operativo» / «Audit migrazione…» (solo portal_gap);
  • Footer — matrix_kind, run id, conteggio righe.

Colonne tabella — tutte le matrici (matrix.columns.mjs)

Colonne tabella — test_coverage (legacy)

Rimosse le intestazioni distinte Feature / Test · gap / Implementazione — ora allineate alla griglia sopra.

Attributi riga HTML

  • data-finding-id — id stabile per merge refresh e API;
  • data-finding-status — stato normalizzato (gap → open);
  • data-finding-sig — firma per stellina ★ su delta refresh;
  • classi finding-resolved, finding-obsolete, docs-fresh.

Marker merge HTML

Ogni tbody è delimitato da commenti <!-- FINDINGS:section_id --> … <!-- /FINDINGS:section_id --> per merge incrementale senza riscrivere l’intera pagina (matrix.refresh.mjs).

8. Pulsante CREA e link Jira

MatriceLabel bottoneLabel summary Jira
portal_gap[Matrix GAP][Matrix-GAP] {titolo finding}
test_coverage[Matrix TEST][Matrix-TEST] {titolo finding}

Flusso CREA:

  1. Click bottone → menu tipo issue (Bug, Story, …);
  2. POST /api/matrix/create-issue (cruscotto) o /api/docs/matrix/create-issue (HOME);
  3. createMatrixFindingIssue — issue Jira + label;
  4. runVeveDbForIssueKey — veve DB su cache (writeTarget: both);
  5. upsert matrix_finding_issue; sync backlog opzionale;
  6. cella HTML aggiornata con link + badge tipo issue.

11. Overlay e percorso DB

Il file SQLite è per overlay istanziato: PROJECT_{PRJ_NAME}/cruscotto_{PRJ_NAME}.db (env CRUSCOTTO_DB_PATH). Matriche e backlog Jira condividono lo stesso DB sull’host PortalAdmin; il product repo (PRODUCT_REPO_PATH) alimenta lo scan gap via grep e gap analysis.
