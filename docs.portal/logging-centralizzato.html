1. Process / Servizi

File: cruscotto.frontend/cruscotto.process.services.manager.mjs (~1700 righe di orchestrazione stack dev).

  • Ring buffer logLines con MAX_LOG_LINES = 3000.
  • Record: { seq, stream, text, at } — stream stdout | stderr | system.
  • Spawn StartUnit e job Prisma: stdio: ["ignore", "pipe", "pipe"].
  • Export: getRepoServicesLogs(cursor), clearRepoServicesLogs().
  • API cruscotto: GET /api/repo/services/logs, DELETE /api/repo/services/logs (svuota buffer).

2. Cursor Agent

File: admin.portal/portal.cursor.agent.manager.mjs.

  • Ring buffer separato, stessa idea (seq + cursor polling).
  • Stream aggiuntivi: assistant, workflow (blocchi END gogo/procedi).
  • Persistenza solo dello stato job su file JSON — non delle righe log.
  • API: GET /api/cursor/agent/logs, POST /api/cursor/agent/clear-logs.

3. Console HOME

File: admin.portal/portal.home.js + #prepare-log in portal.home.html.

  • Buffer client uiConsoleLines (max 100) per click Avvia/Kill/Apri.
  • Tre tail testuali da polling API: prepare (Istanzia), avvio cruscotto, elenco processi Node.
  • refreshConsoleDisplay() concatena sezioni con header --- … ---.
  • Nessun formato strutturato, nessun livello, nessun filtro per sorgente.

Assente oggi

Non esiste lib/portal.log.mjs né altro modulo log condiviso nel repo. I launcher product in lib/portal.launch.dashboard.mjs usano stdio: "ignore" di default — l'output dello stack product non entra in nessun buffer centralizzato.

Gap e limiti

Area Problema
Duplicazione Due implementazioni quasi identiche di ring buffer + cursor (Process vs Agent).
Livelli Nessun debug / info / warn / error; tutto è testo grezzo.
Sorgente Record senza campo source (es. process, agent, home, prepare).
Persistenza Log solo in RAM — riavvio processo = perdita storico.
API Due endpoint distinti; HOME non consuma API log strutturate.
Product stack Spawn con stdio: ignore — visibilità limitata su servizi product.
Filtro Nessuna env tipo PORTAL_LOG_LEVEL per ridurre rumore in dev/CI.

Proposta — lib/portal.log.mjs

Modulo unico esportante createLogger(source) e un bus condiviso:

  • Ring buffer globale (configurabile, es. 5000 righe) con eviction FIFO.
  • Schema uniforme per ogni riga (vedi sotto).
  • Livelli e filtro via PORTAL_LOG_LEVEL (default info).
  • Adapter per sostituire i buffer locali in cruscotto.process.services.manager.mjs e portal.cursor.agent.manager.mjs.
  • API unificata GET /api/logs?cursor=&source=&level= sul server cruscotto (e proxy opzionale da HOME).
  • Persistenza opzionale (fase 2): tail file rotante sotto data/portal.log — non nel MVP backend.
// Uso previsto
import { createLogger } from "../lib/portal.log.mjs";

const log = createLogger("process");
log.info("avvio unità auth", { unitId: "auth" });
log.stderr("errore bind porta 4000");

Schema record

{
  "seq"    : 1842,
  "at"     : "2026-06-23T14:32:01.123Z",
  "source" : "process",          // process | agent | home | prepare | system
  "stream" : "stderr",          // stdout | stderr | system | assistant | workflow
  "level"  : "error",           // debug | info | warn | error
  "text"   : "EADDRINUSE :4000"
}

Compatibilità: i consumer attuali che leggono solo seq, stream, text, at restano validi; source e level sono additive.
