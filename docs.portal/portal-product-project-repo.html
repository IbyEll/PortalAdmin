In una riga: repo (portal) = dove gira PortalAdmin · product repo = dove sta il codice da buildare/testare · project = quale product è selezionato in config (PRJ_* + PROJECT_*)

1. Repo PortalAdmin

Questo repository Git: cruscotto, runner, report, DB SQLite admin.

Simbolo nel codiceFileSignificato
getPortalRoot() admin.portal.lib/portal-paths.mjs Path assoluto della root PortalAdmin/
getPortalDataDir(), data/reports/ portal-paths, reporter Dati locali admin (report JSON/HTML, cache)
cruscotto.database/ SQLite cruscotto DB del tooling — non del product

2. Product repo

Checkout del monorepo applicativo esterno a PortalAdmin: API, web, testScript/, Prisma.

Simbolo nel codiceSignificato
getProductRepoPath() Path assoluto del checkout product
PRODUCT_REPO_PATH (.env) Override path; se vuoto → ../PRJ_REPO
productRoot, REPO_ROOT (in test.catalog) Stessa root product, nomi variabili nei moduli
admin.portal.lib/product.prisma.mjs @prisma/client caricato dal product, non da PortalAdmin
resolveProductSeedPath(productRoot) Path seed Prisma dentro il product

3. Project (profilo config)

Non è un path su disco: è quale product PortalAdmin è configurato per servire, tramite overlay in PROJECT_{name}/.

Simbolo nel codiceSignificato
PRJ_NAME (.env) Nome overlay attivo (PRJ_NAME obbligatorio in .env; es. AdminDashBoard)
resolveProjectOverlayName() Legge PRJ_NAME
getProjectConfig() Oggetto PRJ_* (Jira, DB filename, health URL, …)
PROJECT_JustLastOne/ Overlay: catalogo test, fixture, discovery, runner config
admin.portal.lib/test.catalog.mjs Facade → PROJECT_${name}/test.catalog.${name}.mjs 
// admin.portal.lib/project.config.mjs

  `../PROJECT_${overlayName}/project.config.${overlayName}.mjs`

Confusioni frequenti nel codice

NomeSembra…In realtà
PRJ_REPO «repo del project» Solo nome cartella sibling default del product da project.config (es. JustLastOne → ../JustLastOne, PortalAdmin → ../PortalAdmin)
PRJ_GITHUB_REPO repo PortalAdmin Nome repository GitHub del product
REPO_ROOT in test.catalog root PortalAdmin Root del product (getProductRepoPath())
repo-services-manager un solo repo Gestisce servizi su product + portal
Esempio product esterno: PRJ_NAME=JustLastOne e product repo ../JustLastOne. Esempio PortalAdmin come product: PRJ_NAME=AdminDashBoard e PRODUCT_REPO_PATH sul checkout PortalAdmin. sembrano la stessa cosa. Restano layer distinti: puoi cambiare PRODUCT_REPO_PATH (clone altrove) mantenendo lo stesso PRJ_NAME, o aggiungere PROJECT_MyApp/ prima di avere quel repo clonato.

Layout su disco

PortalAdmin/                    ← getPortalRoot()      [repo admin]
  lib/                            ← codice generico
  admin.portal.lib/test.run.all.mjs
  PROJECT_JustLastOne/            ← overlay project       [PRJ_NAME]
    project.config.JustLastOne.mjs
    test.catalog.JustLastOne.mjs
  data/reports/                   ← output admin

../JustLastOne/                   ← getProductRepoPath() [product repo]
  apps/api/
  apps/web/
  testScript/
  packages/database/

Nuovo product

  1. Checkout del monorepo (product repo) — eventuale PRODUCT_REPO_PATH in .env
  2. Cartella PROJECT_MyApp/ con overlay (project.config.MyApp.mjs, test.catalog.MyApp.mjs, …)
  3. PRJ_NAME=MyApp in .env

Le facade in admin.portal.lib/ (test.catalog.mjs, product.prisma.mjs, run-all.mjs) restano invariate.
