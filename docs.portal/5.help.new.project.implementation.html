Implementazione nuovo progetto

PortalAdmin — overlay PROJECT_{PRJ_NAME} · aggiornato giugno 2026

Vedi anche: Product, project e repo · Config runner overlay · DB cruscotto

Guida operativa per registrare un nuovo product in PortalAdmin: variabili d'ambiente, cartella overlay, file di config e verifiche di avvio. Non modifica codice — descrive cosa creare e dove.

In una riga: imposta PRJ_NAME + PRODUCT_REPO_PATH, crea PROJECT_{Nome}/project.config.{Nome}.mjs e test.catalog.{Nome}.mjs; aggiungi discovery/runner se serve lo stack dev completo.

1. Variabili d'ambiente

Copia .env.example in .env e compila almeno:

# Overlay attivo — obbligatorio (nessun default in codice)
PRJ_NAME=MyApp

# Checkout assoluto del product repo (monorepo da buildare/testare)
PRODUCT_REPO_PATH=C:\dev\MyApp

# Porta cruscotto — opzionale se definita PRJ_DASHBOARD_PORT nel project.config
DASHBOARD_PORT=3999

# HOME multi-istanza (selettore progetti)
PORTAL_HOME_PORT=3990

# Backlog Jira → SQLite (npm run db:sync) — solo se usi cache Jira
JIRA_EMAIL=...
JIRA_API_TOKEN=...
JIRA_CLOUD_ID=...
JIRA_BOARD_ID=...
PRJ_NAME deve coincidere con il suffisso della cartella PROJECT_{PRJ_NAME} (es. PRJ_NAME=JustLastOne → PROJECT_JustLastOne/). Senza PRJ_NAME falliscono getProjectConfig(), discovery, test catalog e cruscotto.

2. Cartella overlay PROJECT_{Nome}/

Tutti i valori specifici del product vivono sotto PortalAdmin/PROJECT_{Nome}/. I moduli in admin.portal.lib/ importano dinamicamente in base a PRJ_NAME.

2.1 Minimo (obbligatorio per HOME cruscotto)

FileExportRuolo
project.config.{Nome}.mjs PROJECT_CONFIG_VALUES Costanti PRJ_*: Jira prefix, GitHub, path DB Prisma, testScript, URL health auth/api/web, porta dashboard, manifest stack. Schema completo in admin.portal.lib/project.config.mjs → PROJECT_CONFIG_KEYS (tutte obbligatorie).
test.catalog.{Nome}.mjs catalogo test Elenco testScript per tab Test del cruscotto. Richiesto da admin.portal.lib/portal.instance.mjs per considerare l'overlay valido in HOME.

2.2 Stack dev (consigliato — product tipo monorepo)

FileExportRuolo
discovery.config.{Nome}.mjs DISCOVERY_CONFIG_VALUES Servizi tab Servizi/Process, path script avvio, extras (friendbot, dashboard), descrizioni UI. Caricato da admin.portal.lib/discovery.config.mjs.
runner.config.{Nome}.mjs RUNNER_CONFIG_VALUES Turbo/build, cleanPaths, envFiles, ordine workspace, devStack (auth, api, web). Usato da cruscotto.runner.stack.config.overlay.mjs.

2.3 Jira e gap analysis (opzionale)

FileRuolo
signals.catalog.{Nome}.mjs PRODUCT_REPO_SIGNALS — mappa issue key → path/test nel product repo. Usato da close-story, pillar matrix, backlog insights.
test.custom.match-fixtures.{Nome}.mjs Fixture test match (re-export da admin.portal.lib/test.match-fixtures.mjs).
test.technical.meta.{Nome}.mjs, test.functional.meta.{Nome}.mjs, … Metadati analisi tecniche/funzionali in dashboard (se previsti).
Pagine HTML / plan sotto PROJECT_{Nome}/ Es. cruscotto.jira.project.tree.html, pillar matrix — solo se serve lo stesso tooling MVP di JustLastOne; non obbligatori per un nuovo product.

3. Contenuto project.config.{Nome}.mjs

Parti da un overlay esistente e adatta i valori:

  • PROJECT_JustLastOne/project.config.JustLastOne.mjs — product monorepo
  • PROJECT_AdminDashBoard/config.project.AdminDashBoard.mjs — tooling su PortalAdmin
ChiaveCosa impostare
PRJ_REPONome cartella sibling (../MyApp se checkout affiancato)
PRJ_NAME / PRJ_SLUGEtichetta UI e slug (kebab-case)
PRJ_JIRA_PREFIXPrefisso issue Jira (JLO, ADMIN, …)
PRJ_GITHUB_OWNER / PRJ_GITHUB_REPORepository GitHub del product

PRJ_AUTH_HEALTH_URL, PRJ_API_HEALTH_URL, PRJ_WEB_OPEN_URLEndpoint health stack dev

4. Due casi d'uso

CasoEsempioOverlay tipico
Product monorepo JustLastOne project + test.catalog + discovery + runner + signals (+ asset Jira/pillar se serve)
Tooling PortalAdmin AdminDashBoard project + test.catalog; PRODUCT_REPO_PATH può puntare a JustLastOne per e2e con stack product reale

5. Registrazione in HOME (opzionale)

Per ordinare le card nella HOME multi-istanza, aggiungi il nome overlay in HOME_PROJECT_ORDER dentro admin.portal.lib/portal.instance.mjs. Porte di fallback in DEFAULT_OVERLAY_PORTS se PRJ_DASHBOARD_PORT non è valida.

6. Avvio e verifica

  1. Crea PROJECT_MyApp/ con almeno project.config e test.catalog.
  2. Imposta .env (PRJ_NAME=MyApp, PRODUCT_REPO_PATH assoluto).
  3. (Consigliato) Aggiungi discovery.config e runner.config copiando da JustLastOne e adattando servizi/path.
  4. Avvia HOME: npm run portal:home (o script equivalente) → seleziona il progetto, oppure prepare: node admin.portal/portal.instance.prepare.mjs --overlay MyApp
  5. Avvia cruscotto sulla porta PRJ_DASHBOARD_PORT (es. npm run admin:dashboard).
  6. (Opzionale) Sync backlog Jira in SQLite: npm run db:sync — richiede credenziali JIRA_*.
  7. (Opzionale) Popola catalogo segnali da branch/piano: node admin.portal.JiraCORE/sync-repo-catalog.mjs --dry-run
Smoke rapido: con PRJ_NAME impostato, esegui node test.smoke/smoke-portal-config.mjs per verificare merge config Jira e segnali.

7. Checklist

  • obbl. .env — PRJ_NAME, PRODUCT_REPO_PATH
  • obbl. PROJECT_{Nome}/project.config.{Nome}.mjs — tutte le chiavi PRJ_*
  • obbl. PROJECT_{Nome}/test.catalog.{Nome}.mjs
  • cons. discovery.config.{Nome}.mjs
  • cons. runner.config.{Nome}.mjs
  • opt. signals.catalog.{Nome}.mjs
  • opt. Credenziali Jira + db:sync
  • opt. Voce in HOME_PROJECT_ORDER
