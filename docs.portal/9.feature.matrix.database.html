Oggi le matrici avanzamento/gap/audit/test vivono in file JSON + HTML con merge incrementale e un piccolo store matrix.finding-issues.json per i link Issue refinement. La proposta sposta righe, snapshot, storico e correlazioni Jira nel DB SQLite del cruscotto (PROJECT_{overlay}/cruscotto_{overlay}.db), mantenendo HTML/JSON come artefatti di export e non più come sorgente di verità.

Problemi da risolvere

  • Storicità incompleta — lo stato obsoleto è gestito via merge JSON; non c’è traccia append-only di quando una voce è sparita dallo scan o è stata risolta.
  • Coerenza Issue refinement — link finding↔Jira in JSON separato; rischio drift rispetto a jira_issue.status e a issue chiuse su Jira ma ancora open in matrice.
  • Doppia sorgente — HTML merge, JSON scan e store link non transazionali; rigenerazione parziale può perdere stati manuali.
  • Multi-matrice — gap unificato, test coverage e futuri audit condividono renderer ma non un catalogo righe centralizzato queryabile (API, cruscotto UI, report).
  • Overlay istanziato — il DB cruscotto è già per overlay (PRJ_NAME); le matrici restano solo su file sotto PortalAdmin host.

Migrazione da JSON

  1. Migration Prisma + seed matrix_catalog da docs.portal.lib/matrix.repo.audit.mjs (AUDIT_CATALOG).
  2. Script matrix.db.import.mjs --from docs.portal/matrix.portal.gap.json + repo-audit-ridondanze-gap.json + matrix.finding-issues.json.
  3. Flag transizione MATRIX_PERSIST=db|json|dual — fase dual scrive DB + JSON; confronto diff in CI.
  4. Deprecare merge HTML (refreshMatrixPageHtml stati issue) → render full da DB preservando issueKey in tabella.
  5. Aggiornare 3.info.cruscotto.db.schema.html con le nuove tabelle.
