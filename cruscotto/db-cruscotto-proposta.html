Database dedicato al cruscotto

Possiamo creare un database dedicato al cruscotto? Immagino una tabella Jira con lo scarico delle issue + info required dal cruscotto in cui diamo una nostra chiave…

Sì, ha senso — e oggi non c’è un DB cruscotto: ogni vista richiama Jira live (fetchJiraBacklog() → JQL paginato + sprint board Agile) e ricalcola tutto in memoria. Un database dedicato risolverebbe latenza, rate limit e la duplicazione degli scan.

Stato attuale

Jira Cloud API  ──►  fetchJiraBacklog()  ──►  JSON in memoria  ──►  backlog / insights / matrice / my-project
                              │
                              └── nessuna persistenza (solo localStorage insight lato browser)

Ogni chiamata scarica ~400+ issue + sprint + board keys. La matrice aggiunge scan repo + git log — il collo di bottiglia resterebbe repo/git, ma Jira non andrebbe più ripetuto 4–5 volte.

Proposta: DB Admin separato dal prodotto

Non metterlo in packages/database/prisma (SQLite dell’app :4000). Meglio un DB locale Admin: 

data/cruscotto.db          ← SQLite dedicato
lib/cruscotto-db/          ← schema + client (Prisma o better-sqlite3)

Motivi: zero impatto su deploy prodotto, schema libero, backup/reset indipendente, ok per dati “tooling”.

Chiave propria vs JLO-xxx

Approccio Pro
UUID interno + jira_key unique FK pulite, storico sync, issue link, pilastri con id stabile
Solo jira_key come PK più semplice, sufficiente se non serve storico

Consiglio: UUID + jira_key unique + tabella sync_run per sapere “questo backlog è di quando”.

Cosa non duplicare nel DB

  • HTML matrice pilastri → resta file generato (o blob opzionale)
  • Insight testuali → possono restare derivati on-read o tabella insight_log se servono storico
  • Piano Working (JLO_WORKING_PLAN) → resta codice/config; in DB solo mapping sprint board (working_plan_sprint_keys)

Migrazione graduale (basso rischio)

  1. Fase A — sync-jira-backlog.mjs + loadJiraBacklog() che legge DB, fallback API se vuoto/stale
  2. Fase B — dashboard-server usa loader unico; fetchJiraBacklog diventa thin wrapper
  3. Fase C — cache repo/git nello stesso DB; matrice legge tutto da lì
  4. Fase D (opz.) — webhook Jira → sync incrementale per key

Trade-off

Pro Contro
Cruscotto veloce e offline-tolerant Serve job sync + gestione stale
Una fonte per backlog/insights/matrice Schema da mantenere allineato a JiraBacklogRow
Base per storico (“com’era il backlog il 14/06”) Duplicazione controllata rispetto a Jira (accettabile per tooling)
Repo/git cacheable → matrice da minuti a secondi Non sostituisce Jira come source of truth operativa

Verdetto: sì, è la direzione giusta. Partirei da SQLite in data/, tabella jira_issue con UUID + jira_key, sync batch che replica oggi fetchJiraBacklog() inclusi i campi derivati (devOrder, tier, …). Il prodotto Prisma resta intoccato.

Passo concreto successivo: schema Prisma minimale + sync-jira-backlog.mjs + refactor di fetchJiraBacklog con flag CRUSCOTTO_DB=1.

Comando veve — 4 regole modulari

Cosa fa il comando veve? Come sono strutturati i 4 step?

veve JLO-xxx è l’orchestratore del backlog grooming: allinea Jira al repo in 4 step, ciascuno con regola e comando standalone. Nessun codice, commit o PR.

Sintassi — grooming completo

  • veve JLO-507
  • veve Story JLO-507 · veve Bug JLO-894 · veve Todo JLO-930

Case-insensitive sulla key. Funziona su Story, Bug o Todo.

Regole Cursor (`.cursor/rules/`)

# File regola Comando standalone Cosa fa
1 JLO-Veve-LeggiJira.mdc leggi jira JLO-xxx Lettura parent, subtask, orphan, key invalide (solo read Jira)
2 JLO-AnalizzaRepo.mdc
adattatore veve: JLO-Veve-AnalizzaRepo.mdc		 analizza repo JLO-xxx Gap analysis repo vs ticket (regola canonica riusabile fuori da veve)
3 JLO-Veve-AggiornaStory.mdc aggiorna story JLO-xxx editJiraIssue description parent (AC, DoD, Stato repo, ordine)
4 JLO-Veve-AggiornaSubtask.mdc aggiorna subtask JLO-xxx Edit / riparent / create subtask (file, dipendenze, ordine N/M)

Orchestratore: JLO-Veve.mdc — veve JLO-xxx esegue 1→2→3→4 in sequenza. Ogni step è invocabile da solo; rispetta i prerequisiti (es. aggiorna story richiede contesto da Leggi Jira + Analizza repo nella stessa sessione).

Comandi utili

veve JLO-507              → grooming completo (1→2→3→4)
leggi jira JLO-507        → solo step 1
analizza repo JLO-507     → solo step 2
aggiorna story JLO-507    → solo step 3
aggiorna subtask JLO-507  → step 4 (tutte le subtask del parent)
aggiorna subtask JLO-524  → step 4 (singola subtask)

Sequenza dettagliata

  1. Leggi Jira — getJiraIssue + JQL figli/orphan; conserva description, stati e gerarchia in sessione. Vietato edit.
  2. Analizza il repo — regola canonica JLO-AnalizzaRepo.mdc + skill jlo-analizza-repo + CLI analyze-repo-keys.mjs; tabella Stato repo (✅/⚠️/❌); ordine subtask consigliato. Vedi sezione Analizza repo centralizzato.
  3. Aggiorna Story — description parent: Obiettivo, Stato repo, AC, DoD, ordine subtask, fuori scope. Vietato transizione Fatto.
  4. Aggiorna Subtask — per ogni figlia: Obiettivo, Stato repo, AC, DoD, file coinvolti, dipendenze, ordine N/M; riparent se parent sbagliato; crea subtask solo se gap chiaro. Riepilogo finale repo vs Jira.

Cosa non fa (veve e step modulari)

Non fa A differenza di…
Branch, commit, PR procedi / gogo
Implementazione codice procedi
Chiudere ticket a Fatto chiudi Story / chiudi Bug
Silenzio chat (blocchi START/END) procedi / gogo
Task «Jira Auto» MCP procedi / gogo / chiudi

Tutti gli step veve usano MCP Jira diretto (user-atlassian, CallMcpTool).

Quando usarlo

Prima di implementare, quando un ticket è vecchio o disallineato rispetto al repo. Dopo veve (o dopo step 1–4), per il codice: procedi Story JLO-xxx.

Esempio

veve JLO-507 → (1) legge parent + subtask → (2) gap repo → (3) edit JLO-507 → (4) JLO-524…616 aggiornate, JLO-525/616 riparentate, ordine 524 → 525 → 616 → 526.

Analizza repo — centralizzato

Come riusare l’analisi repo (veve, procedi, chiudi, review) senza duplicare logica?

Lo step «Analizza repo» è estratto in un modulo condiviso, una CLI, una regola canonica e una skill agente. Veve step 2 usa solo un adattatore sottile (JLO-Veve-AnalizzaRepo.mdc).

Artefatti nel repo

Path Ruolo

scripts/analyze-repo-keys.mjs CLI JSON / markdown per parent o elenco key
.cursor/rules/JLO-AnalizzaRepo.mdc Regola canonica (trigger, aree, output, sessione)
.cursor/skills/jlo-analizza-repo/SKILL.md Skill agente — flusso CLI + deep dive grep/read

CLI

node scripts/analyze-repo-keys.mjs --parent JLO-507
node scripts/analyze-repo-keys.mjs --parent JLO-507 --format md
node scripts/analyze-repo-keys.mjs --keys JLO-524,JLO-525

Output JSON: issues[] con key, esito (ok | partial | absent), symbol (✅⚠️❌), paths, gap, subtaskOrder (con --parent).

Flusso agente

  1. Baseline automatica — CLI --parent o --keys
  2. Deep dive — grep/read sulle aree + description Jira
  3. Merge — arricchisci tabella CLI (overlap story sorella, stub, subtask Fatto)
  4. Output — Stato repo + Issue vs repo + ordine subtask; conserva in sessione

Ordine subtask consigliato: schema → shared → API → web → i18n → test → verifica manuale (suggestSubtaskOrder() nel modulo).
