Ordine config.* e runner.*

PortalAdmin — overlay product, stack dev, discovery cruscotto · aggiornato giugno 2026

Ordine di caricamento (runtime)

Tutto parte da PRJ_NAME in .env (obbligatorio; es. AdminDashBoard o JustLastOne) tramite lib/portal.load.env.mjs:

  1. lib/portal.load.env.mjs
  2. lib/project.config.mjs → import dinamico project.config.{PRJ_NAME}.mjs
  3. lib/discovery.config.mjs → import dinamico discovery.config.{PRJ_NAME}.mjs
  4. runner/runner.config.stack.mjs → import dinamico runner.config.{PRJ_NAME}.mjs
  5. runner/runner.stack.base.mjs (nessun overlay — solo path/npm)
  6. runner/runner.stack.mjs (usa punti 2, 4 e 5)
  7. Entrypoint (start_DEV_Service, start_ALL_Services, …)

discovery.config e runner.config.stack sono paralleli (entrambi dipendono solo da project.config); runner.stack li unisce con runner.stack.base.

lib/config.* — dal generico al product

# File Ruolo
1 project.config.mjs Radice — resolveProjectOverlayName(), getProjectConfig(), costanti PRJ_*
1b project.config.JustLastOne.mjs Dati JLO: repo, Jira, URL health, path DB, manifest
2 discovery.config.mjs → import dinamico discovery.config.{PRJ_NAME}.mjs Discovery cruscotto — merge overlay + PRJ_NAME
2b discovery.config.JustLastOne.mjs → import dinamico discovery.config.JustLastOne.mjs Dati JLO: servizi core, friendbot, path runner, descrizioni tab
— cruscotto.api.documentation.config.mjs A parte — payload OpenAPI documentation (serve-api-documentation, start_API_Documentation); usa portal-paths + manifest, non la catena overlay sopra

runner/* — dal generico all'orchestrazione

# File Ruolo
1 runner.stack.base.mjs Base — run/runTurbo, root, .env, cleanup (senza product)
2 runner.config.stack.mjs Config stack dev — getRunnerConfig(), resolveDevServices(), clean/build order
2b runner.config.JustLastOne.mjs Dati JLO: devStack auth/api/web, daemon, clean paths
3 runner.stack.mjs Orchestrazione — prepare repo, build, Prisma, startDevStack, daemon
4 Entrypoint CLI cruscotto.process.start.service.mjs — un servizio (auth | api | web)
cruscotto.process.start.all.services.mjs — stack completo
cruscotto.process.start.api.documentation.mjs / cruscotto.api.documentation.server.mjs — API Documentation
cruscotto.process.stop.all.services.mjs, cruscotto.process.kill.ports.mjs, … — utility

Regola pratica

Pattern Significato
*.mjs senza suffisso Modulo generico + funzioni resolve* / get*
*.{PRJ_NAME}.mjs Solo dati del product (PROJECT_CONFIG_VALUES, RUNNER_CONFIG_VALUES, …)
runner.stack.* / runner.config.* Stack dev: base (shell), config (resolve), orchestrazione
config.* Identità product + discovery cruscotto
config_api-portal Specializzato API docs, fuori catena overlay
